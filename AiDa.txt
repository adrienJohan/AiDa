# AiDa - Master Architecture & Development Document



## Project Overview



AiDa stands for **Artificial Intelligence Daily Assistant**.



The goal of AiDa is not to be a simple chatbot.



The goal is to build an AI assistant composed of multiple specialized agents that collaborate to help users with:



* Fitness coaching

* Nutrition coaching

* Progress analysis



The user interacts through a natural conversation while AiDa manages memory, onboarding, routing and specialized tasks behind the scenes.

# Design Principles



## Principle 1



Agents do not ask questions.



Agents only:



* Extract information

* Analyze information

* Generate outputs



The Orchestrator is responsible for:



* Asking questions

* Managing conversations

* Routing requests

* Managing workflows



---



## Principle 2



Users can refuse to answer questions.



AiDa must never get stuck in an infinite loop asking for information.



If a user refuses a question:



* The field can be marked as DECLINED

* The workflow continues whenever possible



---



## Principle 3



Long-term memory and short-term memory are different.



Long-term memory:



* Profile information

* Preferences

* Health notes



Stored in database.



Short-term memory:



* Recent conversations

* Temporary situations



Stored in conversation history.



Example:



Permanent:



* Asthma

* Chronic knee pain



Temporary:



* Broken foot this week

* Traveling this weekend

# Session State Architecture



AiDa maintains session state.



Example:



```python

session = {

    "mode": "coach",

    "waiting_for": "workout_equipment"

}

```



Possible modes:



* onboarding

* assistant

* coach

* nutrition

* analysis



Purpose:



Allows AiDa to understand answers such as:



"I have dumbbells."



without rerouting.

# Coach Agent



Status: Designed



Responsibilities:



1. Collect coaching information.

2. Generate workout plans.



Owned fields:



* workout_equipment

* workout_preferences

* health_notes



Workflow:



User requests workout.



Coach Agent checks required information.



If information missing:



Returns missing fields.



Orchestrator asks questions.



When enough information exists:



Generate workout.



Output:



Weekly workout program.



Includes:



* Training days

* Rest days

* Exercises

* Sets

* Reps

* Cardio duration



Future:



Workout adaptation from user feedback.

# Analyst Agent



Status: Designed



Purpose:



Transform stored data into insights.



NOT a reporting agent.



The Analyst must identify patterns.



Input:



* Profiles

* Meals

* Workouts



Examples:



Bad:



"You completed 4 workouts."



Good:



"You are becoming more consistent."

"You consumed 90g protein."

"Protein intake drops on weekends."

* Patterns

* Trends

* Recommendations
